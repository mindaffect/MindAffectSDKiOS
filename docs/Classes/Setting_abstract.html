Setting_abstract

public class Setting_abstract : NSObject

Setting classes can subclass this abstract class to more easily implement the Setting protocol.

Important

subclasses should call valueWillChange and valueChanged at appropriate times.

  • validityCheck

    A closure which checks whether a certain value is a valid value for the setting.

    This closure gets a ‘candidate’ for the setting’s new value and should return nil if that candidate is ‘valid’. If it is not valid, the closure should return an explanation for the user why the value is not valid.

    Attention

    Classes that implement the Setting protocol should perform the validityCheck before they actually change their setting value. They should also not forget to copy the validityCheck in their copy function.

    Whether the setting allows to be undefined.

    Settings cannot be undefined by default.

    Important

    we do not use optionals for representing specific setting type values. Instead in most cases the concept of ‘undefined’ is irrelevant. This property, as well as allowsUndefined, is only used for implementing the ‘Feedback Screen’, where ‘undefined’ should be the default for all ‘settings’. But these ‘settings’ are only used for logging purposes.
